## Completed Tickets

All work before TICKET-0001 was done directly through conversation and recorded in `memory/architecture.md`, `memory/decisions.md`, and `memory/project_memory.md` instead of numbered tickets - don't retroactively invent ticket numbers for that untracked work, since that would create a false impression of a history that didn't actually happen this way. The ticket system starts fresh from TICKET-0001 below.

* **TICKET-0001** (2026-07-06, Bug) - Server could launch on a stale port instead of the one actually configured, because `instances.json`'s stored `gamePort` and the live `PalWorldSettings.ini`'s `PublicPort` could disagree with nothing reconciling them. World Settings' "Game Port" field is now the single place the port is edited; starting a server enforces that port into the ini (self-healing if missing) and syncs it back into `instances.json`.
* **TICKET-0002** (2026-07-06, Bug) - `palworld_settings._get_field`/`_set_field` couldn't see a key that was the first field in `OptionSettings=(...)`, found while building TICKET-0001's self-heal path. Fixed the lookbehind to also accept start-of-string.
* **TICKET-0003** (2026-07-06, Bug) - Neither `RemoteAccessPanel` nor `PortForwardPanel` could show or remove a UPnP port mapping unless *that browser session* had just created it - `forwarded` was local-only React state, never derived from the router's actual mapping. Added `upnp.get_port_mapping()` (`GetSpecificPortMappingEntry`) and exposed real mapping state (including which machine currently holds it) via `/network/upnp/status`, so a mapping from another machine or an earlier session can now be seen and removed - the actual problem reported (router's own UI couldn't remove a UPnP-added entry, and neither could this tool until now).
* **TICKET-0004** (2026-07-06, Enhancement) - Follow-up UI reorg after reviewing the TICKET-0003 build: moved the active server's disk path from Dashboard (visible to all admins) to Super Admin (role-gated); removed the "Game Port" field from World Settings and made Super Admin's "Share With Friends" port field the one and only place to view/edit/save a server's actual port; removed `ServerConnectionPanel` and its mock API entirely (confirmed with the user it was dead, non-functional leftover UI with no real backend, not something to relocate).
* **TICKET-0005** (2026-07-06, Bug) - TICKET-0003's "Remove This Forward" button only showed when the router's `GetSpecificPortMappingEntry` query succeeded - a far less reliably-supported UPnP action than Add/Delete across consumer routers, so it silently never appeared for the user. Made "Forward" and "Remove" both unconditionally available whenever a UPnP router is present, since removal already safely no-ops (UPnP 714) if there's nothing mapped - no condition on detection needed at all.
* **TICKET-0007** (2026-07-06, Bug) - Dashboard CPU% didn't match Task Manager - `psutil.Process.cpu_percent()` is relative to one core, not normalized across all logical cores like Task Manager, so it read inflated by the core count on any multi-core machine. Fixed by dividing by `psutil.cpu_count()` in `process_manager._tree_cpu_ram()`. RAM reporting was checked and found already correct.
* **TICKET-0006** (2026-07-06, Bug) - Kicking a player completed without error but didn't disconnect them. Two real bugs fixed: `show_players()` wasn't stripping whitespace from `ShowPlayers` CSV fields; and, found from the user's own server console log (`User id: steam_<SteamID64>`), `KickPlayer`/`BanPlayer`/`UnBanPlayer` needed that same `steam_`-prefixed ID form, not the bare numeric ID `ShowPlayers` returns. Confirmed fixed by the user against a real kick.
* **TICKET-0008** (2026-07-06, Bug) - Security audit finding: any authenticated user (not just super admin) could read the real RCON `AdminPassword`/`ServerPassword` in plaintext via `GET /api/server-settings`. Now redacted for non-super-admins; only the super admin can view or change these fields. Verified via a real HTTP request through the app.
* **TICKET-0009** (2026-07-06, Bug) - Security audit finding: any authenticated user could register an arbitrary host folder as a "server instance," override the mods path to any folder, or trigger native folder-picker dialogs on the host desktop. Restricted instance import/deploy/remove and mods-path override/browse to super admin only (matches how comparable game-server panels scope this); `InstanceManagerPanel` UI updated to match. Verified via real HTTP requests (403 for a regular admin).
* **TICKET-0010** (2026-07-06, Bug) - Security audit finding: login response timing revealed whether a username existed (instant for unknown users, ~67ms PBKDF2 cost for real ones). Fixed by always running a dummy PBKDF2 comparison for unknown usernames. Measured before/after: timing ratio went from near-zero to ~1.01.
* **TICKET-0011** (2026-07-06, Bug) - Security audit finding: session tokens never expired server-side (only the client-side cookie had a 30-day expiry). Added a matching server-side max age to `session_store`, verified sessions are correctly rejected and pruned once expired.
* **TICKET-0012** (2026-07-06, Enhancement) - Locked the whole Settings tab behind super admin (nav-hidden + route-guarded, same pattern as Super Admin), following on from TICKET-0009. Moved `Ue4ssPanel` out of Settings and into the Mods page (and its file from `components/settings/` to `components/mods/`) so UE4SS management stays available to regular admins. Simplified `InstanceManagerPanel`'s TICKET-0009 role checks back out since they're now unreachable dead code - the page itself is gated.
* **TICKET-0013** (2026-07-06, Bug) - Fresh security audit (inspired by reviewing Crafty Controller's static file handler): `spa_fallback` in `main.py` concatenated a raw, unauthenticated user path onto the frontend directory with no containment check of its own - not currently exploitable (Starlette normalizes `..` out of the URL before the handler runs) but that's incidental framework behavior, not a guarantee. Added an explicit resolved-path containment check, same pattern as `mod_installer._safe_extract`. Verified by calling the function directly with a raw traversal string, bypassing whatever the HTTP layer normalizes.
* **TICKET-0014** (2026-07-06, Bug) - Same audit: `automation.router` was still `_authed` (any admin) after TICKET-0012 locked its only UI entry point behind super admin. Changed to `_super_admin_only` to match. Verified via a real HTTP request (403 for a regular admin).
* **TICKET-0015** (2026-07-06, Enhancement) - Moved "Install From File" mod upload from a conditionally-hidden button on the Mods page to its own section on Super Admin. Already correctly permission-gated on both ends beforehand (backend `require_super_admin`, frontend role check) - purely an organization change, consolidating high-risk actions onto the one page that's entirely reserved for the super admin instead of scattering hidden conditionals across shared pages.
* **TICKET-0016** (2026-07-06, Enhancement) - Added a shared `ManualForwardInstructions` component (name, protocol, external/internal IP, external/internal port) shown when UPnP isn't available, in both the admin-port and game-port forwarding panels. Added `local_ip` to `/network/upnp/status` so the exact LAN IP to enter is shown, not just a generic "forward it manually" message. Verified end-to-end against the real router/network.
* **TICKET-0017** (2026-07-06, Enhancement) - Removed "(any)" from the External IP row. Removed the "Install Mod" manual-entry feature entirely (backend route, API call, frontend button/dialog) - it only ever created an untracked, non-file-backed mod entry indistinguishable from a real one in the UI, confirmed with the user it had no real value as built.

Confirmed this with the user (AskUserQuestion) and pivoted to logging real events instead: server start/stop/force-kill, player join/leave (now logged regardless of the `joinLeaveMessages` toggle, which only gates the in-game broadcast), kick/ban/unban (with who and whom), and scheduled backup/restart outcomes.
